# Release Notes



**버전:** 2026.1.17b

**배포 일자:** 2026년 1월 17일

**주요 요약:** 데이터베이스 연결 안정화, 게시물 좋아요 기능 도입 및 동시성 제어 로직 적용



---



### 1. 신규 기능 (New Features)



#### 게시물 좋아요(Post Like) 시스템 도입



* 사용자가 게시물에 대해 좋아요를 표현할 수 있는 기능을 추가하였습니다.

* **토글(Toggle) 방식 적용:** 단일 API 엔드포인트를 통해 좋아요 등록과 취소가 가능하도록 구현하여 사용자 편의성을 높였습니다.

* **데이터 정합성 확보:** `PostLike` 별도 엔티티를 생성하고, 사용자 식별자(User ID)와 게시물 식별자(Post ID)의 조합에 유니크 제약 조건을 설정하여 중복 등록을 원천 차단하였습니다.



#### 좋아요 카운트 캐싱 및 성능 최적화



* 게시물 상세 조회 및 목록 조회 성능을 향상시키기 위해 `Post` 엔티티 내에 `like_count` 컬럼을 추가하여 카운트를 관리합니다.

* 자주 조회되는 좋아요 여부 확인 기능을 위해 `user_id` 컬럼에 인덱스를 적용하여 쿼리 성능을 최적화하였습니다.



---



### 2. 버그 수정 및 성능 개선 (Bug Fixes & Improvements)



#### 데이터베이스 연결 오류 수정



* **JDBC 연결 문자열 정규화:** Railway 환경 배포 시 발생하던 `JDBCConnectionException`을 해결하였습니다. 기존의 `DATABASE_URL` 환경 변수 대신 개별 환경 변수(Host, Port, DB Name 등)를 조합하는 방식으로 수정하여 연결 안정성을 확보하였습니다.

* **네트워크 지연 시간 단축:** 서버 배포 지역을 미국에서 싱가포르로 이전하였습니다. 이를 통해 국내 사용자 기준 평균 응답 속도를 약 50% 이상 개선하였습니다.



#### 동시성 이슈 해결 (Concurrency Control)



* **원자적 업데이트 적용:** 다수의 사용자가 동시에 좋아요를 누를 때 발생하는 카운트 오차 문제를 해결하였습니다. 애플리케이션 레벨의 연산 대신 데이터베이스의 원자적 연산(`UPDATE ... SET like_count = like_count + 1`)을 사용하도록 수정하였습니다.

* **영속성 컨텍스트 관리:** 벌크 수정 쿼리 실행 후 영속성 컨텍스트와 데이터베이스 간의 불일치를 방지하기 위해 `clearAutomatically = true` 옵션을 적용, 데이터 일관성을 유지하였습니다.



---



### 3. 인프라 및 배포 (Infrastructure & Deployment)



#### 환경 분리 및 비용 최적화



* 단일 PostgreSQL 인스턴스 내에서 운영(`prod`) 및 개발(`dev`) 데이터베이스를 논리적으로 분리하여 관리하도록 설정하였습니다. 이를 통해 데이터 격리 수준은 유지하면서 인프라 운영 비용을 절감하였습니다.

* **Docker 기반 배포:** 전체 애플리케이션 환경을 컨테이너화하여 로컬 환경과 배포 환경 간의 일관성을 확보하였습니다.



---



### 4. 향후 로드맵 (Upcoming Tasks)



* **인프라 이전:** 비용 효율성 극대화를 위해 현재의 사용량 기반 모델에서 AWS Lightsail 고정 비용 모델로의 이전을 계획 중입니다.

* **CI/CD 자동화:** GitHub Actions와 Docker를 연동하여 코드 푸시 시 자동으로 빌드 및 배포되는 파이프라인을 구축할 예정입니다.

* **캐싱 도입:** 좋아요 카운트 등 빈번한 조회가 발생하는 데이터에 대해 Redis 등 메모리 기반 저장소를 활용한 캐싱 전략을 검토하고 있습니다.
